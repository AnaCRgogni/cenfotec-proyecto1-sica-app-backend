## Project overview

Back-end for SICA, an asset control and management web app (Node.js/Express + MongoDB via Mongoose). It exposes a REST API used by a separate front-end to manage organizational **units**, **assets**, **users**, and asset **transfers** between units, including an approval workflow and email notifications.

## Running the project

There is no `package.json` committed (it, `package-lock.json`, and `node_modules/` are gitignored), so there are no npm scripts, tests, or linter configs in the repo. To run the server locally:

```bash
npm init -y   # if package.json doesn't exist locally
npm install express mongoose cors nodemailer
node index.js
```

The server listens on port `8000`. The MongoDB connection string (MongoDB Atlas) and the Gmail SMTP credentials used for outgoing mail are hardcoded in `index.js` and `bl/send_email.js` respectively — there is no `.env`/config layer.

## Architecture

**Entry point**: `index.js` connects to MongoDB, creates the top-level Express `app`, and mounts one sub-router per domain area via `app.use(...)`. Each file under `routes/` independently does `const app = express(); app.use(express.json()); app.use(cors({}))` and exports that mini-app — routers are not built with `express.Router()`, they're full Express apps mounted on the parent app.

**No controller/service layer**: route handlers in `routes/*.js` call Mongoose models in `models/*.js` directly. There's no separate business-logic layer beyond `bl/`.

**Domain models** (`models/`): `units_model.js`, `assets_model.js`, `users_model.js`, `transfers_model.js`. IDs are **not** Mongo ObjectIds — each collection has its own zero-padded sequential string `id` (or `transferId` for transfers), generated by `bl/next_id.js`. `next_id.js` computes the next id by making an HTTP `fetch` call back to the running server itself (`http://127.0.0.1:8000/...`) to read the current max id, so the server must already be up and reachable at that URL for creation endpoints to work.

**Approval workflow**: `assets`, `users`, and `transfers` each carry an `isApproved` boolean. "Normal" routes (`/assets`, `/users`, `/transfers`) generally return/operate on approved records, while the request-review routes (`asset-requests` in `assets_request_route.js`, `user-requests` in `user_requests_route.js`, `transfers-requests`/`transfers_requests` in `transfers_requests_route.js`) surface the unapproved ones for an admin to approve/reject/delete. Note the inconsistent naming across these route files (`asset-requests` singular vs `transfers-requests` vs `transfers_requests`, underscores vs hyphens) — match the existing convention in the specific file you're editing rather than "fixing" it project-wide.

**Role-based scoping (no auth middleware)**: there's no session/JWT auth layer. Instead, many GET routes accept an `id` query param (`?id=<userId>`) identifying the calling user, look that user up via `usersModel.find({ id: queryId })`, and if `user[0].role === 'Encargado de Inventario por Unidad'` scope the query to that user's own `unit`; otherwise the query is unscoped (treated as an admin/global role). This pattern is duplicated per-route rather than centralized — follow it when adding new scoped routes.

**Auth**: `routes/signin_route.js` compares plaintext passwords (`userByEmail.password !== password`) — there is no hashing. `routes/users_route.js` handles signup (auto-generates a random password and creation date, sets `isApproved: false`, `changePassword: true`), password checks, and password updates, also with plaintext comparison.

**Email** (`bl/send_email.js`): uses `nodemailer` with hardcoded Gmail SMTP credentials to send three transactional emails, each with an inline HTML template: new-account password (`sendPasswordEmail`), password-change confirmation (`passChangeSuccessEmail`), and transfer approval/rejection result (`sendTransferResultEmail`). These are invoked from `signin_route.js` (forgot-password), `users_route.js` (password update), and `transfers_route.js` (transfer approval/rejection).

**Search endpoints**: routes like `/assets/search/by-id/:id`, `/units/search/by-name/:name`, `/users/search/by-email/:email` build a `RegExp` directly from the URL param (`new RegExp(`(?=.*${param})`, 'i')`) for partial matching — no escaping of regex special characters.

**Transfers**: creating a transfer (`POST /transfers`) and resolving one (`POST /transfers/:id`) both bundle an asset update in the same request body (`assetUpdatedInfo`) alongside the transfer data, and update the `assetsModel` record for the asset being moved as part of the same handler.
